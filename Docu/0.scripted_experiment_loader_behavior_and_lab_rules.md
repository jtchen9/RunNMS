# Scripted Experiment Loader Behavior and Lab Operation Rules

## Purpose

This document defines the production behavior for scripted experiment loading in the NMS command subsystem. It applies to all three script-loading APIs:

- `POST /cmd/_load_script`
- `POST /cmd/_load_csv`
- `POST /cmd/_load_csv_file`

The goal is to make every scripted mobility experiment start from a deterministic and safe state. Operators should not need to remember manual Redis cleanup steps before running a script.

---

## Supported loader formats

### 1. `/cmd/_load_script`

Input is a JSON body:

```json
{
  "t0": "2026-06-13-10:30:00",
  "items": [
    {
      "scanner": "twin-scout-charlie",
      "t_offset_sec": 0,
      "category": "mobility",
      "action": "mobility.report.location",
      "args": {}
    },
    {
      "scanner": "twin-scout-charlie",
      "t_offset_sec": 75,
      "category": "mobility",
      "action": "mobility.turn_move_turn.forward",
      "args": {"pre_angle": 0.0, "distance_m": 0.10, "post_angle": 0.0}
    }
  ]
}
```

### 2. `/cmd/_load_csv`

Input is a JSON body with `t0` and `csv_text`:

```json
{
  "t0": "2026-06-13-10:30:00",
  "csv_text": "scanner,t_offset_sec,category,action,args_json\ntwin-scout-charlie,0,mobility,mobility.report.location,{}\n"
}
```

### 3. `/cmd/_load_csv_file`

Input is multipart form data:

- `t0`: absolute NMS local start time, format `YYYY-MM-DD-HH:MM:SS`
- `csv_file`: uploaded `.csv` file

CSV columns are always:

```csv
scanner,t_offset_sec,category,action,args_json
```

---

## Common behavior for all three loaders

All three loaders now use the same mobility script-run preflight before accepting mobility rows.

If the submitted script contains no mobility rows, the mobility preflight is skipped.

If the submitted script contains one or more mobility rows, the loader performs the following production preflight before scheduling any row:

1. Clear the global mobility stop latch:
   - `nms:mobility:experiment_stop_state_json`

2. Clear old scheduled mobility commands:
   - `nms:mobility:cmd`

3. For each whitelisted scanner participating in the submitted mobility script:
   - Clear its immediate robot command stream, e.g. `nms:cmd:twin-scout-charlie`.
   - Reset mobility state to `s0idle`.
   - Set `robot_safety_state = NORMAL`.
   - Set `stop_experiment = false` and clear `stop_reason`.
   - Clear retry/collision/busy/exec-failure counters.
   - Clear pending sequence and outgoing command preview.
   - Clear stale planned pose and previous command mirror fields:
     - `planned_location_json`
     - `last_planned_command_action`
     - `last_planned_command_args_json`
     - `last_planned_command_source`
     - `last_planned_command_updated_at`

4. Enforce the mobility initialization rule:
   - For each scanner, the earliest mobility row must be `mobility.report.location`.
   - If not, the loader rejects the script with HTTP 400.

The loader response includes a `preflight` object showing whether mobility preflight was applied and which scanners were reset.

## Required mobility script structure

Every mobility script for every scanner must start with:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-charlie,0,mobility,mobility.report.location,{}
```

A movement command must come only after the location command has enough time to complete. For early lab testing, use at least 60–90 seconds after `mobility.report.location` before the first movement.

Example short smoke test:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-charlie,0,mobility,mobility.report.location,{}
twin-scout-charlie,75,mobility,mobility.turn_move_turn.forward,"{""pre_angle"":0.0,""distance_m"":0.10,""post_angle"":0.0}"
```

---

## Meaning of `mobility.report.location`

`mobility.report.location` is not a movement command. It is a location precondition command.

Production behavior should be:

1. Issue `mobility.report.location` to the robot.
2. Wait for the robot mobility report.
3. Solve AprilTag true location.
4. Save `true_location_json`.
5. Align `planned_location_json = true_location_json`.
6. Return to `s0idle`.
7. Do not enter correction movement for a pure location report.

This prevents the script-run from chasing stale planned poses from a previous experiment.

## Lab operation safety rules

### Rule 1: A mobility script must start with location

Every scanner participating in a mobility script must begin with `mobility.report.location`. Movement is not allowed as the first mobility row.

Reason: movement needs a fresh true pose and a planned pose aligned to that true pose.

### Rule 2: One script-run owns the mobility scheduler

Do not load a second mobility script while one is already running. The loader clears `nms:mobility:cmd` before loading a new mobility script, so loading a new script intentionally replaces the old scheduled mobility plan.

### Rule 3: A powered-off robot must be removed from the physical scene immediately

The dynamic obstacle map ignores stale/inactive robots after the configured TTL. Therefore a powered-off robot cannot be trusted to remain represented as an obstacle. If a robot is powered off, physically remove it from the test area.

### Rule 4: Do not leave inactive robots in the test area

Inactive robots may not be included in the dynamic restriction zone. They can become unmodeled physical obstacles.

### Rule 5: Keep the restriction map conservative but correct

If the restriction map is known to be incorrectly drawn, do not run autonomous correction near that area. Either fix the map or move the test to a known safe region.

### Rule 6: Use short smoke tests before long scripts

Before any long experiment, run a short script:

1. `mobility.report.location`
2. one short forward move, e.g. `distance_m = 0.10`

Only proceed to longer scripts after the smoke test confirms that the loader, scheduler, robot polling, and location precondition path work.

### Rule 7: Do not manually lift the robot and continue the same script-run as if nothing happened

If the robot is lifted to avoid collision, the physical pose and NMS planned pose may diverge. Stop the experiment, move the robot to a safe location, and start a new script-run from `mobility.report.location`.

### Rule 8: Use future `t0`

Always set `t0` in the future, not in the past. Recommended: current NMS local time + 30–60 seconds.

### Rule 9: Leave enough time between mobility rows

During early validation, use generous spacing:

- At least 60–90 seconds after `mobility.report.location` before first movement.
- At least 120–180 seconds between movement rows if correction is enabled.

After system timing is characterized, the spacing can be reduced.

### Rule 10: Treat `s7stopped` and the global stop latch as hard stops

If `s7stopped` or `nms:mobility:experiment_stop_state_json.stop = true` appears, no operator should wait for the script to continue. The next script load must reset these states through production preflight.

## Operator checklist

Before loading a script:

1. Confirm all participating robots are powered on, physically present, and in safe regions.
2. Remove powered-off or inactive robots from the scene.
3. Ensure the first mobility row for each scanner is `mobility.report.location`.
4. Set `t0` in the future.
5. Start with a short smoke test before long experiments.

After loading a script:

1. Verify loader response `status = ok`.
2. Verify `preflight.mobility_preflight = ok` if mobility rows are present.
3. Verify `bad_rows = 0` and `skipped_not_whitelisted = 0` for intended robots.
4. Monitor the robot during execution.
5. If manual intervention is required, stop the experiment and restart from location precondition.

---

## Failure behavior

The loader should reject a mobility script with HTTP 400 if any participating scanner's earliest mobility row is not `mobility.report.location`.

A rejected script should not partially enqueue mobility rows.

A loaded script should never depend on a previous run's stale planned pose, old global stop latch, or old scheduled mobility commands.
