🔧 2. Updated State Machine (Replace Entire Final Section)
Full State Machine (current design)
S0_IDLE
  ↓ (script command arrives)
S5_COMPUTING_CORRECTION
  ↓
S6_ISSUING_COMMAND
  ↓
S1_WAITING_REPORT
  ↓
   ├─ correct report → S2_EVALUATING_POLICY
   ├─ irrelevant report → stay S1
   └─ timeout → S4_WAITING_LOCATION_RETRY

S2_EVALUATING_POLICY
  ↓
   ├─ MOBILITY_BUSY → stay S2 (wait timer)
   ├─ ok / no_tag → S3_SOLVING_TRUE_LOCATION
   ├─ recoverable error → S4_WAITING_LOCATION_RETRY
   └─ stop → S7_STOPPED

S3_SOLVING_TRUE_LOCATION
  ↓
   ├─ solved → S5_COMPUTING_CORRECTION
   ├─ cannot solve but allowed → propagate → S5
   └─ cannot solve and not allowed → S4

S4_WAITING_LOCATION_RETRY
  ↓
   ├─ issue mobility.location.report → S1
   └─ retry exhausted → S7

S5_COMPUTING_CORRECTION
  ↓
   ├─ no correction needed → S0_IDLE
   ├─ correction needed → S6
   └─ unsafe → S7

S6_ISSUING_COMMAND
  ↓
   └─ always → S1_WAITING_REPORT

S7_STOPPED
  ↓ (manual reset / admin reset)
S0_IDLE

🔧 3. S1 Design (NEW — add as dedicated section)

S1 — Waiting for Report

Purpose
    Wait for the exact report corresponding to last issued command

Three scenarios
    1. Correct report
        cancel S1 timer
        go to S2
    2. Irrelevant / outdated report
        ignore
        remain in S1
        timer unchanged
    3. Timeout (no report)
        dangerous condition
        increment busy_count
        go to S4
        issue mobility.location.report

Concurrency design
    All S1 exits go through:
        process_s1_event()
    Protected by:
        per-scanner Redis lock

This prevents:
    report vs timeout race
    double transition

Timer rule
    S1 timer is started only when entering S1
    never started by:
        report arrival
        timeout callback

🔧 4. S2 MOBILITY_BUSY Design (NEW)

Behavior
    When robot returns:
    last_error_code = "MOBILITY_BUSY"

Policy:
    1. Increment busy_count
    2. Check total unexpected-event sum
    3. If limit reached → S7
    4. Otherwise:
        set s2_entry_reason = "busy_retry_timer"
        start retry timer (e.g. 5 sec)
        stay in S2

Timer callback
    When timer fires:
        re-enter S2
        check marker
        resend previous command
        start S1 timer
        go to S1

Key design decision
    NO sleep in NMS
    NO lock needed
    ONLY one Redis marker controls flow

🔧 5. Unexpected Event Policy (NEW)

Old design (removed)
    Separate limits:
        busy
        collision
        exec_fail

❌ Problem:
    allowed multiple different failures without stop

New design
    
Define:
"""
unexpected_event_sum =
    busy_count +
    collision_veto_count +
    exec_fail_count
"""

Rule:
    if sum ≥ threshold → S7

Additional rule
    if sum > 0:
        disable true-location propagation

🔧 6. True Location Handling (UPDATED)

Sources
    AprilTag solve → authoritative
    Propagation → fallback

Propagation rule
    Allowed only if:
        no unexpected event occurred
    Otherwise:
        must go to S4 retry

Important design decision
    NO propagation in:
        S0
        S6
    propagation ONLY happens in:
        S3


==============
🔧 7. Planned vs True Separation (IMPORTANT)

Planned location
    updated ONLY in S0
    represents script intent
    never modified by execution

True location
    updated by:
        AprilTag solve
        propagation

Key rule
    Planned = intent
    True = reality
    Never mix the two

🔧 8. S4 Retry Policy (UPDATED)

When used
    missing report (S1 timeout)
    recoverable errors

Behavior
    issue mobility.location.report
    go to S1
    loop until:
        success → S3
        retry exhausted → S7

Important update
    NO_TAG_VISIBLE is NOT an error
    it flows through:
        S2 → S3 → propagation
