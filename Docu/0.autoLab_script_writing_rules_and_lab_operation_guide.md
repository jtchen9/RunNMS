# AutoLab Script Writing Rules and Lab Operation Guide

## Purpose

This document defines the rules for writing and registering scripted AutoLab experiments.

It is intended for remote experiment users, tutorial videos, the future XLSM script-authoring template, CommonCheckers offline validation, lab operators, and NMS `/cmd/_load_csv_file` behavior.

The goal is simple:

> A script that passes the script-writer checker should follow the same rules that the lab operator and NMS use when registering the experiment.

This document is therefore the rule contract. The XLSM file is only a helper tool for producing valid CSV scripts.

---

## 1. AutoLab scripting workflow

The recommended workflow is:

```text
Human script writer
    ↓
XLSM template or manual CSV editing
    ↓
experiment CSV file
    ↓
CommonCheckers offline validation
    ↓
lab operator registers CSV through /cmd/_load_csv_file
    ↓
NMS schedules commands
    ↓
robots / APs execute recognized commands
```

The CSV file is the machine-friendly exchange format. The XLSM file is the human-friendly authoring interface.

---

## 2. Active experiment registration endpoint

Only one endpoint registers scripted experiments:

```text
POST /cmd/_load_csv_file
```

The older endpoints are not part of the active contract:

```text
POST /cmd/_load_script    deprecated / removed
POST /cmd/_load_csv       deprecated / removed
```

This avoids several inconsistent script-loading paths.

### `/cmd/_load_csv_file` input

The endpoint receives multipart form data:

```text
t0        absolute NMS local start time
csv_file  uploaded .csv file
```

`t0` format:

```text
YYYY-MM-DD-HH:MM:SS
```

Example:

```text
2026-07-21-16:30:00
```

The uploaded CSV filename is used as the experiment id.

Example:

```text
M2_test.csv  →  experiment_id = M2_test
```

---

## 3. CSV script contract

Every experiment CSV uses these columns:

```csv
scanner,t_offset_sec,category,action,args_json
```

| Column | Meaning |
|---|---|
| `scanner` | Target robot, AP, or device name. |
| `t_offset_sec` | Command time offset in seconds relative to `t0`. |
| `category` | Command category, such as `mobility`, `traffic`, `scan`, `ap_meta`, or another supported category. |
| `action` | Command action understood by the target subsystem or device. |
| `args_json` | JSON object containing command arguments. Use `{}` when there are no arguments. |

Example:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-delta,0,mobility,mobility.report.location,{}
twin-scout-delta,180,mobility,mobility.move,"{""x_m"":5.0,""y_m"":2.0}"
```

### `args_json` rule

`args_json` must be valid JSON and must decode to a JSON object.

Valid:

```json
{}
```

```json
{"x_m": 5.0, "y_m": 2.0}
```

Invalid:

```json
[]
```

```text
x=5.0,y=2.0
```

## 4. Role of the XLSM template

The XLSM template is a human-friendly script-writing tool.

It should help users:

- choose scanners from known robot/AP lists,
- choose known commands from a command catalog,
- fill command arguments with fewer JSON mistakes,
- generate valid CSV,
- run or prepare CommonCheckers validation,
- reduce typing mistakes.

The XLSM template does not define the rules. This document defines the rules.

The XLSM template should export a CSV that follows the contract in this document.

---

## 5. Role of CommonCheckers

CommonCheckers is the offline script checker shared by script writers and lab operators.

It checks rules that can be checked before the experiment starts, such as:

- CSV structure,
- valid JSON in `args_json`,
- known script-level mobility command vocabulary,
- required mobility initialization rows,
- timing between mobility rows,
- maximum single `mobility.move` distance,
- site workspace bounds,
- static restriction map crossing,
- charging/rest zone crossing,
- bump guard crossing,
- macro start-pose rules,
- planned robot-to-robot clearance.

CommonCheckers must be usable outside the NMS runtime codebase, because script writers only receive the script-authoring package.

---

## 6. Role of `/cmd/_load_csv_file`

The NMS registration endpoint is the lab-side gate.

It must check rules that depend on registration time or NMS state, such as:

- `t0` must be in the future relative to NMS local time,
- the uploaded file must be a `.csv`,
- the filename must provide a non-empty experiment id,
- scanner whitelist and registration status,
- current experiment registry state,
- no conflicting registered experiment,
- final command enqueueing,
- experiment registry record creation.

The intended final behavior is that `/cmd/_load_csv_file` also runs the same CommonCheckers rule set before accepting the experiment. This keeps the script writer and lab operator under the same rule contract.

## 7. Time and scheduling rules

### Rule 7.1: `t0` must be in the future

`t0` is not checked by offline CommonCheckers because the script writer may create the CSV long before the lab operator registers it.

The NMS checks this at `/cmd/_load_csv_file`.

Rule:

```text
t0 must be later than current NMS local time.
```

Recommended operator practice:

```text
t0 = current NMS local time + 30 to 60 seconds
```

### Rule 7.2: `t_offset_sec` is relative to `t0`

For each row:

```text
execute_at = t0 + t_offset_sec
```

Example:

```text
t0 = 2026-07-21-16:30:00
t_offset_sec = 180
execute_at = 2026-07-21-16:33:00
```

### Rule 7.3: Leave enough time for mobility state-machine progress

Mobility commands are not simple one-shot commands. A movement may involve:

```text
script command
robot execution
location report
pose solve
S5 evaluation
optional correction
final report
```

Frozen timing rules:

```text
First movement after mobility.report.location for the same robot: at least 60 seconds.
Between any two moving mobility commands globally: at least 180 seconds.
```

The 180-second global rule applies across all robots, not only within one robot.

---

## 8. General command categories

The CSV format is not mobility-only.

AutoLab scripts may contain commands for multiple subsystems, such as:

```text
mobility
traffic
scan
ap_meta
```

and later additional robot/AP command categories.

The NMS often acts as a scheduler and relay. Some commands are understood mainly by the robot or AP command parser. Therefore the complete non-mobility command catalog must be derived from the current robot-side and AP-side parsers.

Until the command catalog is fully frozen, the rule is:

```text
Known commands should be selected from the command catalog.
Custom or pass-through commands must be clearly marked as such in the authoring tool.
```

For mobility commands, the vocabulary is already strict and is defined below.

---

## 9. Mobility command vocabulary

Script writers must use semantic mobility commands.

Allowed script-level mobility actions:

```text
mobility.report.location
mobility.move
mobility.in2out
mobility.out2in
```

Blocked low-level mobility actions:

```text
mobility.turn
mobility.turn_move_turn.forward
mobility.turn_move_turn.backward
```

Reason:

- script writers describe the intended movement,
- NMS converts semantic movement into the correct low-level robot command,
- low-level commands may depend on current pose, heading convention, safety state, correction policy, and site-specific behavior.

---

## 10. Required mobility initialization rule

For each scanner participating in mobility:

```text
The earliest mobility row for that scanner must be mobility.report.location.
```

Example:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-delta,0,mobility,mobility.report.location,{}
twin-scout-delta,180,mobility,mobility.move,"{""x_m"":5.0,""y_m"":2.0}"
```

Invalid example:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-delta,0,mobility,mobility.move,"{""x_m"":5.0,""y_m"":2.0}"
```

Reason:

- movement needs a fresh true pose,
- planned pose must be aligned to the true pose at the start of the script,
- stale planned pose from a previous experiment must not be reused.

---

## 11. Meaning of `mobility.report.location`

`mobility.report.location` is not a movement command.

It is a location precondition command.

Expected behavior:

1. NMS issues `mobility.report.location` to the robot.
2. Robot reports current AprilTag observations.
3. NMS solves the robot's true location.
4. NMS saves `true_location_json`.
5. NMS aligns `planned_location_json = true_location_json`.
6. Mobility state returns to `s0idle`.
7. No correction movement is generated for a pure `mobility.report.location`.

This prevents the script from chasing stale planned poses.

---

## 12. `mobility.move` rule

`mobility.move` is the normal script-level movement command.

The preferred form is absolute target coordinates:

```json
{"x_m": 5.0, "y_m": 2.0}
```

Alternative supported form is relative delta:

```json
{"dx_m": 0.5, "dy_m": 0.0}
```

Do not mix absolute and relative arguments in the same row.

Invalid:

```json
{"x_m": 5.0, "y_m": 2.0, "dx_m": 0.5}
```

Optional heading may be supported as:

```json
{"x_m": 5.0, "y_m": 2.0, "heading_deg": 270}
```

The final low-level turn/move/turn details are generated by NMS.

### Maximum single move distance

Frozen rule:

```text
Each script-level mobility.move must be no longer than 3.0 m.
```

Longer routes must be split into multiple shorter `mobility.move` rows.

Example:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-delta,0,mobility,mobility.report.location,{}
twin-scout-delta,180,mobility,mobility.move,"{""x_m"":5.0,""y_m"":2.0}"
twin-scout-delta,360,mobility,mobility.move,"{""x_m"":7.5,""y_m"":2.0}"
```

Remember that moving rows must still obey the global 180-second spacing rule.

---

## 13. Bump-crossing macros

Normal `mobility.move` must not cross the bump guard zone.

Crossing the bump is allowed only through site-defined macros:

```text
mobility.in2out
mobility.out2in
```

Macro behavior is site-specific and defined in:

```text
sitemap/<site>/script_authoring/config/macro_policy.json
```

The macro checker validates:

- macro action is known,
- macro arguments are allowed or forbidden by policy,
- planned robot pose is close enough to the required macro start point,
- macro target is inside workspace bounds,
- planned robot clearance is maintained.

For current macro v1, macros accept no custom arguments. Use `{}`.

Example:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-delta,0,mobility,mobility.report.location,{}
twin-scout-delta,180,mobility,mobility.move,"{""x_m"":9.0,""y_m"":4.35}"
twin-scout-delta,360,mobility,mobility.in2out,{}
```

## 14. Site geometry rules

Site-specific script-authoring configuration is stored under:

```text
sitemap/<site>/script_authoring/config/
```

For DemoRoom this includes:

```text
macro_policy.json
bump_guard_zones.json
zone_policy.json
path_policy.json
safety_policy.json
```

The checker validates planned mobility paths against:

- workspace bounds,
- `restriction_map.npy`,
- charging/rest zones,
- bump guard zones,
- macro start and target rules,
- planned robot-to-robot clearance.

### Restriction map convention

The restriction map is a 2-D NumPy array.

Convention:

```text
0 = free
nonzero = restricted
row = floor((y_m - origin_y_m) / resolution_m)
col = floor((x_m - origin_x_m) / resolution_m)
```

Out-of-map points are treated as restricted.

## 15. Robot-to-robot safety radius

DemoRoom robot-to-robot clearance is configured in:

```text
sitemap/DemoRoom/script_authoring/config/safety_policy.json
```

Current DemoRoom rule:

```json
"robot_safety_radius_m": 0.60
```

CommonCheckers uses this value for offline planned robot-to-robot clearance.

Runtime S5 dynamic obstacle checking uses the matching runtime copy in `config.py`:

```python
MOBILITY_ROBOT_RESTRICT_RADIUS_M = 0.60
```

For DemoRoom, these values must remain synchronized. If one changes, update the other in the same commit.

Reason:

- another robot is a physical obstacle,
- another robot may block AprilTag visibility,
- localization and walking error require margin,
- the value was increased after physical boundary testing.

## 16. Lab operation safety rules

### Rule 16.1: One experiment registration owns the scheduler

Do not register a second experiment while one is already registered or running.

The NMS experiment registry is the authority. If an old experiment remains registered, delete it before registering a new one.

### Rule 16.2: Powered-off robots must be removed from the physical scene

The runtime dynamic obstacle map ignores stale/inactive robots after a TTL.

Therefore, a powered-off robot cannot be trusted to remain represented as an obstacle.

If a robot is powered off, physically remove it from the test area.

### Rule 16.3: Do not leave inactive robots in the test area

Inactive robots may become unmodeled physical obstacles.

Remove inactive robots from the experiment area unless they are intentionally represented by current policy/data.

### Rule 16.4: Keep the restriction map conservative but correct

If the restriction map is known to be wrong, do not run autonomous movement near that area.

Either fix the map or move the experiment to a known safe region.

### Rule 16.5: Use short smoke tests before long scripts

Before any long experiment, run a short script that verifies:

1. command registration,
2. robot polling,
3. `mobility.report.location`,
4. one simple `mobility.move`,
5. S5 follow-up behavior.

Example smoke-test pattern:

```csv
scanner,t_offset_sec,category,action,args_json
twin-scout-delta,0,mobility,mobility.report.location,{}
twin-scout-delta,180,mobility,mobility.move,"{""x_m"":5.0,""y_m"":2.0}"
```

Use a target that is known to be safe in the current site map.

### Rule 16.6: Do not manually lift the robot and continue the same script-run

If a robot is lifted, pushed, blocked, or manually moved, the physical pose and NMS planned pose may diverge.

Stop the experiment, move the robot to a safe known location, and restart from `mobility.report.location`.

### Rule 16.7: Treat `s7stopped` and the global stop latch as hard stops

If a robot reaches:

```text
s7stopped
```

or the global stop latch is set:

```text
nms:mobility:experiment_stop_state_json.stop = true
```

do not wait for the script to continue automatically.

Treat this as a hard stop and investigate before the next run.

## 17. Failure behavior

A script should be rejected before registration if it violates frozen preflight rules.

Examples of rejection reasons:

- CSV missing required columns,
- bad `args_json`,
- unsupported script-level mobility action,
- first mobility row is not `mobility.report.location`,
- first movement too soon after `mobility.report.location`,
- moving commands too close together globally,
- single `mobility.move` longer than 3.0 m,
- planned path outside workspace bounds,
- planned path crosses `restriction_map.npy`,
- normal `mobility.move` crosses bump guard zone,
- macro start pose mismatch,
- planned robot path too close to another robot.

A rejected script must not partially enqueue mobility rows.

A loaded script must not depend on stale state from a previous run.

## 18. Script writer checklist

Before submitting a script:

1. Use the standard CSV columns:
   ```text
   scanner,t_offset_sec,category,action,args_json
   ```
2. Make sure `args_json` is valid JSON object text.
3. For each mobility robot, start with `mobility.report.location`.
4. Put the first movement at least 60 seconds after `mobility.report.location`.
5. Keep every pair of moving mobility commands at least 180 seconds apart globally.
6. Keep every single `mobility.move` at or below 3.0 m.
7. Use `mobility.in2out` or `mobility.out2in` for bump crossing.
8. Do not use low-level mobility commands in scripts.
9. Run CommonCheckers before giving the CSV to the lab operator.
10. For remote experiments, confirm that the lab operator has the correct site package and current policy files.

---

## 19. Lab operator checklist

Before registering an experiment:

1. Confirm the CSV came from the approved script-writing flow or passed CommonCheckers.
2. Confirm all participating robots/APs are intended for the experiment.
3. Remove powered-off or inactive robots from the physical scene.
4. Confirm the restriction map and site policy files are current.
5. Confirm no old experiment is still registered.
6. Choose `t0` in the future, typically current NMS local time + 30-60 seconds.
7. Register through:
   ```text
   POST /cmd/_load_csv_file
   ```
8. Check the registration response:
   ```text
   status = ok
   bad_rows = 0
   skipped_not_whitelisted = 0
   preflight = ok
   ```
9. Monitor execution.
10. If manual intervention is needed, stop the experiment and restart from the location precondition.

## 20. Tutorial-video outline

A useful YouTube tutorial sequence can follow this structure:

### Video 1: Running a remote AutoLab experiment

1. What AutoLab is.
2. What robots/APs do.
3. What the web page shows.
4. How an experiment is registered.
5. What `t0` means.
6. What the operator watches during execution.

### Video 2: Writing a basic CSV script

1. Explain the five CSV columns.
2. Show a simple non-mobility row.
3. Show `args_json`.
4. Show common JSON mistakes.
5. Export or save the CSV.

### Video 3: Writing a mobility script safely

1. Start every robot with `mobility.report.location`.
2. Wait 60 seconds before first movement.
3. Use `mobility.move`, not low-level commands.
4. Keep each move within 3.0 m.
5. Keep moving rows 180 seconds apart globally.
6. Run CommonCheckers.

### Video 4: Site rules and bump-crossing macros

1. Explain restriction map.
2. Explain charging/rest zones.
3. Explain bump guard zone.
4. Explain `mobility.in2out` and `mobility.out2in`.
5. Explain why normal `mobility.move` must not cross the bump.

### Video 5: Troubleshooting failed validation

1. Read the validation report.
2. Fix bad JSON.
3. Fix timing errors.
4. Fix unsafe path errors.
5. Fix macro start-pose errors.
6. Re-run validation before submission.
